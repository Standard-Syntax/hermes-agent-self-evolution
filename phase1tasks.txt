Review summary

Phase 1 is partially implemented, but not complete enough to call production-ready. The repo has the right core files for Phase 1: config, constraints, dataset generation, external importers, scoring, skill wrapping, and the evolve_skill CLI. The README already labels Phase 1 as implemented, but the PLAN defines Phase 1 as done only when at least one skill improves by ≥10%, benchmark regression is within 2%, the diff is human-sensible, and the pipeline can be reused for any skill.  ￼  ￼

The biggest issue is that Phase 1 currently looks like a prototype path, not a fully gated evolution pipeline.

Main gaps to fix first

1. Constraint validation is probably wrong for skills.
    evolve_skill.py validates skill["body"] as artifact type "skill", but _check_skill_structure() expects full YAML frontmatter with name and description. That means the baseline/evolved body can fail structure checks even when the full SKILL.md is valid.  ￼  ￼
2. --run-tests exists but is not actually used as a gate.
    ConstraintValidator.run_test_suite() exists, but evolve_skill.py does not call it after generating the evolved skill.  ￼  ￼
3. The README promises PR-style deployment, but the code only writes output files.
    The repo says the pipeline produces a best variant through constraint gates and PR review, but current Phase 1 output is saved under output/<skill>/<timestamp>/; there is no branch creation, patch application, diff summary, or PR body generation yet.  ￼  ￼
4. The SkillModule needs proof that GEPA is truly optimizing the skill text.
    The wrapper stores skill_text as a plain module attribute and passes it as an input field to a predictor. Current DSPy GEPA docs describe GEPA as evolving textual components of DSPy programs through predictor traces and reflection, usually via dspy.GEPA(...).compile(student=..., trainset=..., valset=...); this repo should add an integration test proving the optimized program actually mutates the skill instructions instead of returning the baseline unchanged.  ￼  ￼
5. The optimizer API should be verified.
    Current DSPy docs show parameters like max_metric_calls and reflection_lm; the repo currently uses max_steps=iterations, so Phase 1 should include an API-compatibility check against the installed DSPy version.  ￼  ￼

Slice P1-002 — Fix skill parsing and constraint semantics

Files

evolution/skills/skill_module.py
evolution/core/constraints.py
tests/skills/test_skill_module.py
tests/core/test_constraints.py

Changes

Separate these concerns:

full SKILL.md structure validation
body-only size validation
body-only growth validation
frontmatter YAML validation

Do not call this:

validator.validate_all(skill["body"], "skill")

as if the body is the full skill file.

Use one of these designs instead:

validator.validate_skill(
    full_text=skill["raw"],
    body_text=skill["body"],
    baseline_body_text=None,
)

or:

validator.validate_all(skill["raw"], "skill_file")
validator.validate_all(skill["body"], "skill_body")

Also replace the current frontmatter string check with yaml.safe_load() because "name:" in text[:500] is too weak.

Done when

Tests prove:

- valid full SKILL.md passes
- body-only markdown passes body checks
- missing frontmatter fails only structure validation
- evolved body can be reassembled and then validated as a full SKILL.md

Slice P1-006 — Replace weak metric with feedback-rich scoring

Files

evolution/core/fitness.py
tests/core/test_fitness.py

Changes

skill_fitness_metric() currently uses simple keyword overlap. That is fine for a smoke test, but too weak for Phase 1 completion.

Create two modes:

fast_metric        — cheap deterministic smoke metric
judge_metric       — LLM-as-judge with structured feedback
gepa_metric        — returns score plus textual feedback when trace is available

GEPA performs best when the metric provides useful feedback, not just a scalar score; DSPy’s GEPA docs explicitly describe feedback-bearing metrics as part of the optimizer’s design.  ￼

Done when

Holdout evaluation can output:

{
  "task_input": "...",
  "baseline_output": "...",
  "evolved_output": "...",
  "baseline_score": 0.62,
  "evolved_score": 0.78,
  "judge_feedback": "..."
}
